Target Constraints

Decretum supports 84 compiler targets. The syntax stays the same with event and proc blocks, data declarations, and control flow, but each target has its own registers, instruction set, word size, and constraints. You write target specific programs using a shared syntax. This means you can target anything from a 4 bit Intel 4004 to a 64 qubit quantum circuit, but the code itself is still architecture aware.

See the compiler backends page for the complete list with CLI commands, output formats, and architecture details.

bios16

Uses direct 16 bit machine encoding and BIOS interrupt services. Limited to 1 MB address space with real mode segment and offset addressing. Use BIOS helpers (builtin.*) for text output, keyboard input, cursor control, disk I/O, and boot level routines. Output is a 512 byte boot sector with a CHS disk loader that reads a stage2 kernel payload from disk.

Portable and win64 Path

Compiles to Decretum bytecode and executes through the Rust runtime engine. The runtime supports 51 opcodes, 16 registers, string operations, file I/O, and the DCRNG random generator. The compile-pe command wraps bytecode into a native PE executable with an embedded runtime stub. No NASM or external assembler is required.

Direct Machine Code Path

Each architecture backend generates raw machine code for its target. The code runs directly on the hardware. Some targets impose specific constraints: MIPS and SPARC have delay slots after branches. ARM Cortex-M requires a vector table at address 0x0000. The 8 bit backends have limited register sets and address spaces. The quantum backends produce circuit descriptions rather than machine code. Every backend runs a peephole optimization pass before encoding.

Language Level Control Flow

All targets support if, else, endif, while, and endwhile constructs. These compile to native compare and jump sequences at the source level before reaching the backend. Nested constructs are supported.

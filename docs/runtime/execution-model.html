Execution Model

Decretum has three execution paths that share the same parser and produce the same event driven control flow model.

Compilation Pipeline

  1. Parse source into a Program structure with target, entry event, data declarations, and code blocks.
  2. Pass the parsed program to the target specific backend.
  3. bios16 path: Encode directly to 16 bit x86 machine code. Generate a 512 byte boot sector with CHS disk loader and a stage2 kernel payload.
  4. portable or win64 path: Compile to a 51 opcode bytecode format. Optionally wrap the bytecode in a PE executable with a runtime stub.
  5. Direct machine code path: Each backend is a full two pass assembler. Pass one lowers instructions to binary. Pass two resolves labels and writes the output format.

Event Driven Flow

Event dispatch is explicit through emit, which fires a named event and transfers control. Procedure calls are explicit through call and return with ret. The entry directive specifies which event runs at program startup. Events can call procedures, emit other events, or use control flow constructs for branching and looping.

Portable Bytecode Runtime

The portable runtime is a register based virtual machine implemented in src/portable.rs. It maintains:

  • 16 general purpose 64 bit registers
  • A memory map associating string names with 64 bit integer values
  • A string storage map for dynamic string variables
  • A frame stack for call and return tracking
  • Signed and unsigned comparison flags for conditional jumps
  • A 50 million instruction execution guard to prevent infinite loops

DCRNG Random Generator

The runtime includes a custom 320 bit ARX PRNG called DCRNG. It maintains a 320 bit state across 5 internal 64 bit words and uses a 2 round ARX construction with rotation constants 29, 17, 11, 7, 23, 19, 13, and 5. A Weyl sequence extends the period beyond 2^320. The generator supports raw 64 bit output, bounded ranges, and is used by the random and random_range operations.

Bytecode Format

Bytecode files use the .dcb format with the following structure:

Magic: "DCB2" (4 bytes)
Version: 2 (2 bytes, little endian)
Entry event name: length-prefixed string
Integer data count: u32, then name-value pairs
String data count: u32, then name-string pairs
Block count: u32, then for each block:
  kind (u8: 0=event, 1=proc)
  name (length-prefixed string)
  opcode count (u32)
  encoded opcode stream

Opcodes are variable length. Each opcode starts with a single byte identifying the operation, followed by operands encoded as type-tagged values. The runtime decodes opcodes sequentially and dispatches to the appropriate handler.

Direct Machine Code Execution

Direct machine code backends produce native binaries for the target architecture. These run directly on the hardware, in an emulator, or within a hypervisor. The BIOS16 backend produces a bootable disk image for x86 real mode. The UEFI backend produces a PE32+ image for UEFI firmware. Other backends produce raw binaries for flashing, loading, or linking.
